## 🔔 Latest round — 2026-07-24: Fix intermittent "video won't play in the mockup" (only the thumbnail shows) (verify green, 168 tests)
User: some uploaded videos never play in the mockups — you see the thumbnail but it doesn't move; some work, some don't, no obvious pattern. **Diagnosed with evidence, not guesses, and touched ZERO user data.**
- **Read-only probe of the real files** (Management API → parse the blob → probe URLs → detect codec from the bytes, since ffprobe isn't installed): all 7 real videos are **healthy H.264 (avc1) + AAC (mp4a)**, HTTP 200, CORS `*`, immutable — every browser can decode them. Their `moov` atom sits at the END of the file (not web-optimized), so they take ~1–2.4s to start. (1 legacy item has `url:""` from a pre-fix aborted upload — that one correctly shows only its thumbnail.)
- **In-browser repro** (injected the exact old play path): in isolation all 7 play fine. So it's not the files/codec/CORS — it's **`MockupVideo`'s play logic**. The old code tried **unmuted** `play()` first, and if either the unmuted OR the muted retry was interrupted (a re-render / slide switch during the ~2s these trailing-moov files buffer), it called `setPlaying(false)` and **permanently gave up** — leaving a frozen poster. That's the intermittent "thumbnail but no video".
- **The fix — a self-healing player** (`components/media/MockupVideo.tsx`): starts **muted** (muted autoplay is the only kind browsers never block → playback is guaranteed), retries `play()` on every readiness event (`loadeddata`/`canplay`/`canplaythrough`), never permanently gives up, does one clean `load()` retry on a real error, and **upgrades to sound on the first `playing` event** (unmuting a playing element needs no gesture) — so the sound-on default you liked is preserved, just a beat later. A manual mute via the pill now sticks (won't be re-unmuted). Verified in-browser against the real `.mov`: played, advancing, readyState 4, ended up **unmuted**, no error.
- **Data safety:** this is pure playback logic — it never writes, rewrites, or deletes any media/URL. Existing user videos and files are 100% untouched. The separate legacy `url:""` item is unaffected (still shows its saved thumbnail).
- Files: `components/media/MockupVideo.tsx`. verify=0 errors, 168 tests; browser-verified.

## Done (stage 48 — Video posters: instant thumbnails, no more white/gray boxes, sound-on mockups) ✓ verify green (97 tests)
- **Diagnosis (measured against the live data, not guessed):** read-only Management API queries listed the user's stored videos, then range-requests on the files themselves showed every one is a **QuickTime-brand container (`ftypqt`) with its `moov` atom at the END** — absent from the first 300KB, present in the last. Codec is fine (`avc1`/H.264 + `mp4a`/AAC). Consequence: a browser cannot paint a single frame, or even report `videoWidth`, until the WHOLE file (5–12MB) has downloaded. Measured in a real browser on their 5MB clip: `<video preload="metadata">` first painted at **1808ms**.
